# Internationalization — next-intl

Every app is localized with **next-intl** using the "locale-as-root-segment"
model: routes live under `[locale]`, translations live in `messages/*.json`, and
navigation goes through locale-aware helpers so links always carry the current
locale. Default locale is **`vi`**, with **`en`** as the second.

> Files: `apps/<app>/i18n/{routing,navigation,request}.ts`,
> `apps/<app>/messages/{vi,en}.json`, and the `[locale]` route segment. The
> locale gate also runs in `proxy.ts` ([auth.md](./auth.md)).

---

## 1. The pieces

| File                    | Role                                                        |
| ----------------------- | ----------------------------------------------------------- |
| `i18n/routing.ts`       | declares locales, default, prefix strategy                  |
| `i18n/navigation.ts`    | locale-aware `Link`, `useRouter`, `usePathname`, `redirect` |
| `i18n/request.ts`       | loads the right `messages/*.json` per request               |
| `messages/{vi,en}.json` | the translations, keyed by namespace                        |
| `proxy.ts`              | runs next-intl's middleware to resolve/redirect locale      |
| `next.config.ts`        | `createNextIntlPlugin("./i18n/request.ts")` wires it in     |

## 2. Routing config

`apps/<app>/i18n/routing.ts`:

```ts
import { defineRouting } from "next-intl/routing"

export const routing = defineRouting({
  locales: ["vi", "en"],
  defaultLocale: "vi",
  localePrefix: "as-needed",
})

export type Locale = (typeof routing.locales)[number]
```

- **`localePrefix: "as-needed"`** — the default locale (`vi`) has **no** URL
  prefix (`/dashboard`), non-default locales are prefixed (`/en/dashboard`). This
  keeps the primary-market URLs clean.
- **`Locale`** is exported as a union type — use it wherever you type a locale
  param, never a bare `string`.

---

## 3. Navigation — always use the locale-aware helpers

`apps/<app>/i18n/navigation.ts`:

```ts
import { createNavigation } from "next-intl/navigation"
import { routing } from "./routing"

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing)
```

**Import these from `@/i18n/navigation`, never from `next/link` or
`next/navigation`:**

```tsx
import { Link, useRouter, redirect } from "@/i18n/navigation" // ✅
// import Link from "next/link"                                // ❌ drops the locale
```

They preserve the active locale automatically — `<Link href="/dashboard">` on an
`en` page resolves to `/en/dashboard`. Using the raw Next helpers silently sends
users to the wrong locale. This is exactly what the auth forms and app shell use.

## 4. Loading messages per request

`apps/<app>/i18n/request.ts`:

```ts
import { getRequestConfig } from "next-intl/server"
import { hasLocale } from "next-intl"
import { routing } from "./routing"

export default getRequestConfig(async ({ requestLocale }) => {
  const requested = await requestLocale
  const locale = hasLocale(routing.locales, requested)
    ? requested
    : routing.defaultLocale
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  }
})
```

`hasLocale` validates the incoming segment against the allow-list and falls back
to the default — an unknown `/xx/...` won't crash, it degrades to `vi`. Messages
are dynamically imported so only the active locale's bundle loads.

## 5. Reading translations — server vs client

Same keys, two entry points depending on component type:

```tsx
// Client component ("use client")
import { useTranslations } from "next-intl"
const t = useTranslations()
<h1>{t("auth.loginTitle")}</h1>
```

```tsx
// Server component / generateMetadata
import { getTranslations } from "next-intl/server"
const t = await getTranslations()
export async function generateMetadata() {
  return { title: t("nav.dashboard") }
}
```

Scope to a namespace to shorten keys:

```tsx
const t = useTranslations("auth")
<h1>{t("loginTitle")}</h1>          // → messages.auth.loginTitle
<p>{t("noAccount")}</p>
```

## 6. The messages file

`apps/<app>/messages/en.json` (and the matching `vi.json`) — grouped by
namespace:

```json
{
  "common": { "signIn": "Sign in", "email": "Email", "password": "Password" },
  "auth": {
    "loginTitle": "Welcome back",
    "loginSubtitle": "Sign in to your workspace to continue.",
    "noAccount": "Don't have an account?"
  },
  "nav": { "dashboard": "Dashboard" }
}
```

Rules:

- **`vi.json` and `en.json` must have identical key trees.** A key present in one
  and missing in the other renders the key string in the wrong locale. Keep them
  in lockstep when you add a namespace.
- **Group by feature namespace** (`common`, `auth`, `nav`, `settings`) so scoped
  `useTranslations("settings")` works.
- **Arrays are supported** — `brandPoints` in the auth messages is an array read
  with `t.raw("auth.brandPoints")` when you need the list.

## 7. Static rendering — `setRequestLocale`

For a page/layout to render statically, call `setRequestLocale` with the segment
locale before reading any translations:

```tsx
import { setRequestLocale } from "next-intl/server"

export default async function Layout({
  params,
  children,
}: {
  params: Promise<{ locale: string }>
  children: ReactNode
}) {
  const { locale } = await params
  setRequestLocale(locale) // enables static generation for this subtree
  // …
}
```

Skip this and the segment falls back to dynamic rendering. The locale `layout.tsx`
(which owns `<html lang={locale}>`) is where it belongs.

## 8. Interpolation & plurals

```json
{
  "welcome": "Hi {name}",
  "items": "{count, plural, =0 {No items} one {# item} other {# items}}"
}
```

```tsx
t("welcome", { name: user.name })
t("items", { count: list.length })
```

Use ICU plural syntax rather than concatenating strings by hand — it's the only
way that reads correctly across languages.

## 9. Worked example — a localized nav entry for a new page

Adding `/reports` to the sidebar (ties into
[system-architecture.md §11](../system-architecture.md)):

**1. Add the label to both message files** — same key in each:

```jsonc
// messages/en.json → "nav": { …, "reports": "Reports" }
// messages/vi.json → "nav": { …, "reports": "Báo cáo" }
```

**2. Read it in the sidebar** (`components/app-sidebar.tsx`, client):

```tsx
const t = useTranslations("nav")
<SidebarLink href="/reports">{t("reports")}</SidebarLink> // Link from @/i18n/navigation
```

**3. Title the page** via `generateMetadata` (server):

```tsx
export async function generateMetadata() {
  const t = await getTranslations("nav")
  return { title: t("reports") }
}
```

User-facing chrome is translated; **sample/demo data may stay inline English**
per [code-standards](../code-standards.md) — don't spend effort translating throwaway
fixture text.

## 10. Do / Don't

**Do**

- Import `Link`/`useRouter`/`redirect` from `@/i18n/navigation`.
- Keep `vi.json` and `en.json` key trees identical.
- Scope `useTranslations("namespace")` to shorten keys.
- Call `setRequestLocale(locale)` in layouts/pages that should render statically.
- Use ICU interpolation/plurals, not string concatenation.

**Don't**

- Don't import from `next/link` or `next/navigation` in localized routes — the
  locale is lost.
- Don't hardcode user-facing chrome; put it in `messages/*`.
- Don't let the two message files drift out of sync.
- Don't type a locale as `string` — use the exported `Locale` union.
